## 结论摘要

这批事项可以先分成三类实现：

1. 普通协同模板发起：用印申请单、请假申请单、出差申请单、差旅费审批报销单、业务结算单、餐费审批报销单。它们都能从个人常用模板区定位到普通协同新建页，外层发起壳基本一致。
2. 会议模块发起：新建会议不在协同模板区，来自会议模块导航入口，字段和按钮结构不是普通协同模板，需要单独执行器。
3. 待补入口事项：补签申请单未在个人常用模板区命中；消息流显示该事项类型存在，但本轮没有拿到发起入口。后续需要通过“更多/全部应用/HR”或菜单搜索补采。

接收处理不要按“事项类型 = 一套代码”硬拆。更合理的是：

- 协同类接收先共用工作流接收引擎，运行时检查当前待办详情是否暴露 `ContinueSubmit`、`Archive` 等动作。
- 会议接收走已有的 `meeting.reply` 引擎。
- 事项类型只负责提供业务名称、匹配规则、字段画像、可验证状态和安全边界。

## 实现建议

对外命令建议保持事项化，而不是暴露底层原子动作：

```powershell
python -m bscli.cli.main --home .bscli oa matter matrix --matter "用印申请单" --include launch,received
python -m bscli.cli.main --home .bscli oa matter start preflight --matter "请假申请单" --field ...
python -m bscli.cli.main --home .bscli oa matter receive preflight --keyword "用印" --intent approve --opinion "已阅"
```

内部实现建议分层：

1. `matter_registry`：维护事项名称、同义关键词、模板匹配规则、模块类型、验证状态。
2. `launch_profile`：读取模板或导航入口，生成字段画像、按钮画像和风险边界。
3. `receive_profile`：从当前待办优先、已办历史补充，记录可观察动作和验证状态。
4. `executor_binding`：把事项意图绑定到已治理执行器，例如 `collaboration.launch.save_draft`、`collaboration.receive.continue_submit`、`meeting.reply`、`meeting.start`。
